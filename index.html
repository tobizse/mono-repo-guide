Mono-Repo Architecture & Developer Guide

TypeScript Playwright automation framework with multi-brand support

Introduction▾

A mono-repo is a single Git repository containing multiple logically distinct packages. Instead of 3 separate repos (shared framework, API tests, web tests), everything lives together with clear boundaries.

Why pnpm Workspaces?

  • No phantom dependencies — pnpm does NOT hoist all `node_modules` into one flat tree. If `@s2/api` imports something only `@s2/web` declared, it fails immediately — not silently in CI.
  • `workspace:*` protocol — links packages locally without publishing to npm
  • Single install — one `pnpm install` at root wires everything up
  • `--filter` flag — run commands scoped to a single package

Package Structure▾

Three core packages with strict dependency direction:

packages/
  core/    @s2/core   — brand-agnostic framework
  api/     @s2/api    — API test package
  web/     @s2/web    — Web test package
Dependency rule: @s2/core has no dependencies. @s2/api and @s2/web depend on @s2/core but never on each other.

packages/core Layout

packages/core/
  api/         → BaseAPI, api-paths.ts, brand-api.ts (BrandAPI singleton)
  config/      → playwright configs, global-setup, project builder
  data/        → credentials.json (AES-256-GCM encrypted)
  decorators/  → @Step, @Log
  dto/         → UserDto (credential roles)
  fixtures/    → base-fixtures, browser-state paths
  plugins/     → viptalk-reporter
  types/       → Brand, Environment, TestTags enums
  utils/       → loaders, locator-helpers, logger, test-helper
  web/         → BasePage, BaseComponent, BrandLocators type
  environments.yaml  → All brand configs

packages/api Layout

packages/api/src/
  api/         → Domain API classes (Auth, Account, Game, Payment, etc.)
  assertions/  → Per-domain assertion classes (CommonAssert + 6 domain classes)
  fixtures/    → api-fixture.ts (sole fixture registration point)
  tests/       → *.api.spec.ts test files
  types/       → API-specific types
  utils/       → builders, test helpers

packages/web Layout

packages/web/src/
  actions/     → Action helpers, strategies (HomePageSetupStrategy)
  capabilities/ → BrandCapabilities interface
  fixtures/    → web-fixtures.ts, worker-fixtures.ts
  ui/          → pages/, components/, brands/{brand}/
  utils/       → helpers, locator-helpers
  tests/       → *.web.spec.ts test files, setup/browser-state.setup.ts

TypeScript Configuration▾

File Purpose
`tsconfig.base.json` Shared compiler options (target, strict, Stage 3 decorators)
`packages/*/tsconfig.json` Per-package: local `paths`, includes, references
`tsconfig.json` (root) IDE + `tsc --noEmit`: all aliases across all packages
Keep in sync: Path aliases must be defined in both per-package tsconfig (for builds) and root tsconfig (for IDE).

Path Aliases

Alias Resolves to Usage
`@s2/core` `packages/core` Both api and web
`@api/*` `packages/api/src/*` API package only
`@web/*` `packages/web/src/*` Web package only
`@web/actions/*` `packages/web/src/actions/*` Web package only
`@web/capabilities/*` `packages/web/src/capabilities/*` Web package only
Alias naming: `@web/*` in an API file = wrong; `@api/*` in a web file = wrong. The naming makes violations obvious.

Browser Setup & Auth States▾

When you run a `web:*` project, Playwright automatically runs its dependency project `setup:browser-state-{code}-{env}` first.

What it does

  1. Dialog suppression — calls `BrowserSetupActions.suppressDialogs()`, which cycles the page to suppress startup popups and saves localStorage state
  2. Auth states (parallel) — saves storage state files for test users (fresh, commission roles)

Run setup in isolation

BRAND=OXBET ENV=preprod pnpm --filter @s2/web exec playwright test \
  --project="setup:browser-state-BC159111-preprod"

Writing Tests▾

API Tests

Create the spec file in `packages/api/tests/` named `{domain}.api.spec.ts`:

import { APITest as test, expect, state } from '@api/fixtures/api-fixture';
import { Tags } from '@s2/core/types/test-tags.types';
import { AuthAssert } from '@api/assertions/auth-assertion';

test.describe('Authentication API', { tag: [Tags.Smoke, Tags.API] }, () => {
  test.use({ storageState: state.unAuth });

  test('should login successfully', async ({ authAPI, sharedTestUsers }) => {
    const creds = sharedTestUsers.getCommission();
    const response = await authAPI.login({ username: creds.username, password: creds.password });
    AuthAssert.loginSuccess(response, creds.username);
  });
});

Mandatory API test rules

  • Import `test` as `APITest` from `@api/fixtures/api-fixture`
  • Tag all describe blocks with the `Tags` enum (never raw strings)
  • Set storage state on every describe block: `state.auth` or `state.unAuth`
  • Generate test data **inside** test callbacks (never at module/describe level)
  • Use `sharedTestUsers` for credentials — don't call `CredentialsLoader`

Web Tests

Web tests are native Playwright `.web.spec.ts` files — no BDD, no step definitions.

import { test, expect } from '@web/fixtures/web-fixtures';
import { Tags } from '@s2/core/types/test-tags.types';

test.describe('Login', { tag: [Tags.Smoke, Tags.Web] }, () => {
  test.use({ storageState: state.unAuth });

  test('should login successfully', async ({ loginPage, homePage, sharedTestUsers, brandCapabilities }) => {
    test.skip(
      brandCapabilities.loginVariant === 'not-supported',
      'Login not supported for this brand'
    );

    const creds = sharedTestUsers.getCommission();
    await loginPage.goto();
    await loginPage.header.inputUsername.fill(creds.username);
    await loginPage.header.inputPassword.fill(creds.password);
    await loginPage.header.btnLogin.click();

    await expect(homePage.header.iconUser).toBeVisible();
  });
});

Web Architecture

*.web.spec.ts
  → pages (*.page.ts)      — expose components as @Log() getters
    → components (*.component.ts) — expose locators only
      → brands/{brand}/*.locators.ts — CSS strings or byRole() helpers

Page object rules

  • Extend `BasePage`; pass full `BrandLocators` to constructor
  • Expose components as `@Log()` getters
  • Compound interactions as `@Step()` methods
  • No locators directly on pages — they belong in components
  • No assertions on pages
  • No `async` methods (enforced by pre-commit hook)

Component object rules

  • Extend `BaseComponent`; always implement `get root(): Locator`
  • `get root()` MUST use `this.rootLocator('key')` (not `this.locator()` — infinite recursion)
  • Child elements use `this.locator('key')` — auto-scoped to root
  • Parameterized locators use `this.locatorWith('key', param)`
  • Expose elements only — no action methods (`.fill()`, `.click()`)

Adding a New Brand▾

Adding a brand requires updates in 7 places.

1. Runtime config

File: `packages/core/environments.yaml`

NEWBRAND:
  code: 'BC000000'
  preprod:
    baseURL: 'https://dev05.newbrand.com'
  prod:
    baseURL: 'https://newbrand.com'

2. Brand enum

File: `packages/core/types/brands.types.ts`

export enum Brand {
  NEWBRAND = 'NEWBRAND',
}

3. Web locators

File: `packages/web/src/ui/brands/newbrand/newbrand.locators.ts`

export const NewbrandLocators: BrandLocators = {
  loginPage:  { root: 'body' },
  homePage:   { root: 'body' },
  headerComponent: {
    root:          'body header',
    inputUsername: byRole('textbox', { name: 'Tên đăng nhập' }),
    inputPassword: byRole('textbox', { name: 'Mật khẩu' }),
    btnLogin:      byRole('button',  { name: 'ĐĂNG NHẬP' }),
  },
  heroBannerComponent: {
    root: '.hero-banner',
  },
};
Required sections: `loginPage`, `homePage`, `headerComponent`, `heroBannerComponent`. Optional: `loginForm`, `loginModal`, etc.

4. Web messages

File: `packages/web/src/ui/brands/newbrand/newbrand.messages.ts`

export const NewbrandMessages: BrandMessages = {
  login: {
    errorInvalidCredentials: 'Tên đăng nhập hoặc mật khẩu không đúng',
  },
};

5. Register in fixtures

File: `packages/web/src/utils/helpers/fixtures-helpers.ts`

const BRAND_LOCATORS: Partial> = {
  [Brand.NEWBRAND]: NewbrandLocators,
};

const BRAND_CAPABILITIES: Record = {
  [Brand.NEWBRAND]: { loginVariant: 'header-inline' },
};

6. Credentials

File: `packages/core/data/credentials.json`

pnpm run data:decrypt
# Add entries (passwords with enc: prefix)
pnpm run data:encrypt

7. CI configuration

File Action
`.github/config/test-config.json` Add to `brands` array
`.github/workflows/test-api-prod.yml` Add to `workflow_dispatch` choices
`.github/workflows/test-api-preprod.yml` Add to `workflow_dispatch` choices
`scripts/run-brands.sh` Add to `ALL_BRANDS` array

Pre-commit Hooks▾

All staged `.ts` files run through these checks:

Hook Blocks Fix
`test.only` / `describe.only` Focused tests in commit Remove `.only`
Missing describe tags `test.describe()` without `{ tag: [...] }` Add `{ tag: [Tags.Smoke] }`
Plain-text passwords `credentials.json` without `enc:` prefix Run `pnpm run data:encrypt`
`waitForTimeout(` no comment Arbitrary waits Add `/** Usage: */`
`force: true` no comment Force-clicks Add `/** Usage: */`
Core → api/web imports `packages/core/**` importing `@api/*` or `@web/*` Move code to core
ESLint/Prettier lint-staged violations Run `pnpm run lint:fix && pnpm run format`
POM purity `async` in components/pages Move logic to action classes

Common Pitfalls▾

Pitfall Problem Fix
Core imports `@api/*` or `@web/*` Pre-commit blocks; circular reference Move code to core
`@web/*` import in API file Compile error Move code to core
Alias in only one tsconfig IDE works, build fails Update both per-package and root tsconfig
`test.only` in committed code Pre-commit blocks; CI fails Remove `.only`
Test data at module/describe level Race conditions in parallel Generate inside test callbacks
Missing `get root()` on component TypeScript error Every `BaseComponent` needs `get root()`
`this.locator()` in `get root()` Infinite recursion Use `this.rootLocator('key')`
`page.getByRole(...)` in brand map Evaluated at construction, not query time Use `byRole(...)` helper
Phantom dependencies Works locally, breaks in CI Declare deps in package's own `package.json`
Assertions in pages/components Violates separation of concerns Assertions go in spec files only
